## I. NECESSITY

The serial operational framework requires separate candidacy, specialized
determinations, final admission, validation, certification, integration,
register insertion, Manifest release, and activation actions. The framework has
made governance an end in itself and delays the documentary work it exists to
protect.

The Repository still requires canonical identity, provenance, declared scope,
limitations, uncertainty, reproducibility, passing technical checks, and an
explicit human authorization event. It does not require a chain of offices and
records to repeat the same approval.

This Amendment establishes a two-lane system: a streamlined routine lane and a
protected lane for constitutional or materially high-risk changes.

## II. AUTHORITY BASIS AND EFFECTIVE ORDER

1. Document 000012 Section XII.3 bars CAG-000000005 from evading the founder-
   ratified separation of powers without a distinct higher-authority act.
2. The Founder Operational Authorization Simplification Ratification is that
   limited higher-authority act.
3. A draft pull request or unmerged branch creates no constitutional effect.
4. Upon authenticated founder merge, the founder ratification takes effect
   first and this Amendment takes effect immediately afterward in the same
   coherent repository state.
5. This Amendment operates prospectively and does not cure or invalidate any
   earlier act.

## III. PRESERVATION AND CURRENT RESOLUTION

1. Documents 000005, 000008, 000009, and 000012 remain preserved as admitted

5. No Candidate Cognitive Component Register is required.

## IV. TWO-LANE GOVERNANCE

Every proposed repository change shall use exactly one of the following lanes.

### A. ROUTINE OPERATIONAL LANE

A change belongs in the routine lane when it applies existing definitions and
safeguards, preserves canonical history, and has no protected effect under
Section IV.B.

The routine lane requires:

1. a pull request or equivalent merge unit identifying the intended actions;
2. recoverable documentary basis and provenance;
3. declared canonical identifiers and native classes where applicable;
4. declared scope, intended use, limitations, uncertainty, and exclusions;
5. all required repository records, register entries, ledger assignments,
   Manifest changes, and implementation changes in one coherent unit;
6. passing required CI checks; and
7. authenticated repository-owner merge.

The owner merge is the single final authorization event. If the pull request
expressly declares them, that merge may authorize admission, validation,
certification where required, integration, Manifest inclusion or release, and
activation together.

Routine actions need not be split across multiple pull requests and do not need
separate authority-establishment actions or duplicate decision records.

### B. PROTECTED GOVERNANCE LANE

A change belongs in the protected lane when it:

1. changes foundational or constitutional governance;
2. creates, removes, or changes an ontology class, identifier family,
   predicate semantic, or controlled-vocabulary definition;
3. changes documentary evidence rules, epistemic classifications, or
   constitutional safeguards;
4. deletes canonical evidence, rewrites protected history, proposes identifier
   reuse, or otherwise creates material irreversibility;
5. changes credentials, permissions, security boundaries, or an externally
   consequential production action;
6. seeks to represent interpretation or inference as documentary fact; or
7. is expressly designated protected by the repository owner.

The protected lane requires explicit owner approval and review proportionate to
the identified risk. A protected change may still be reviewed and authorized in
one coherent pull request. Separate role records are required only when the
owner or an unrepealed higher rule expressly requires them for that change.

## V. EVIDENCE IS NOT AUTHORITY

1. CI checks, schemas, tests, validation reports, and specialized reviews are
   evidence supporting a decision.
2. Passing CI does not prove historical or philosophical truth.
3. CI failure blocks a routine merge unless the owner expressly moves the
   change to protected review and records the exception and its reason.
4. An automated system may prepare evidence and propose a change; it may not
   merge or authorize its own proposal.
5. Human authorization is supplied by the authenticated repository-owner merge.

## VI. EFFECT ON DOCUMENT 000005 AND AMD-000000002

For routine native-object admission:

1. Candidate Object means only an unmerged proposed object. No separate
   candidacy decision or Candidate Register entry is required.
2. Ontology, identifier, predicate, and vocabulary conformity may be tested by
   automated checks and recorded in the pull request or object metadata.
3. A separate VAL record from every specialized authority is not required when
   the proposal merely conforms to existing definitions.
4. Specialized-authority review remains required in the protected lane when a
   governed definition changes.
5. CAG-000000006 may review escalated or ambiguous admissions but is not a
   mandatory gate for a routine owner-authorized admission.
6. A standalone DDR is optional unless the change needs a durable explanation
   not adequately preserved by the pull request, commit, and object audit
   history.
7. An ADMITTED outcome is complete when the merged repository state contains
   the declared canonical object, required identifier assignment, references,
   lifecycle and audit data, and passing CI evidence.

The requirements for identifier uniqueness, class legibility, provenance,
referential integrity, register consistency, ledger consistency, and audit
history remain mandatory.

## VII. EFFECT ON DOCUMENT 000008

1. Role separation is functional rather than a requirement to create a distinct
   authority, record, person, or pull request for every stage.
2. One person may perform several functions when each action and its evidence
   remain explicit.
3. The Engine or an LLM may prepare a pull request, validation package, or
   Manifest update but may not authorize or merge its own proposal.
4. Repository learning may proceed through the routine lane without formal
   candidacy.
5. A production run remains pinned to the exact repository commit, Manifest,
   source set, and projection selected at run initialization.

Document 000008 Section IV.5 shall therefore be read as:

> Future learning occurs through an explicit, tested repository change whose
> declared lifecycle actions are authorized by owner merge and take effect only
> in later version-pinned runs.

## VIII. EFFECT ON DOCUMENT 000009

1. Preserved, validated, certified, integrated, and active remain distinguishable
   descriptive states. They may be advanced together by one routine merge.
2. Candidate Cognitive Component is optional and confers no special authority.
3. A Candidate Register is optional and is not a prerequisite to validation,
   certification, integration, Manifest inclusion, or activation.
4. Validation records may be structured files, CI results, pull-request
   evidence, or a combination appropriate to the risk.
5. Separate certification is mandatory only for a protected change or when the
   owner expressly requires it. For routine changes the owner's merge may
   certify the declared scope based on the recorded evidence.
6. A separate integration DDR is optional. Inclusion in the Cognitive Memory
   Register and the production Manifest in the same merged unit records
   integration.
7. A separate Manifest-release authority or release decision is not required.
   A Manifest included in the owner-merged unit is released by that merge.
8. Inclusion of a component in the production Manifest activates it for later
   runs within the declared scope. Active runs remain frozen.
9. Development Mode remains nonproduction and clearly labeled.

The minimum acquisition test is replaced by the following questions:

1. What exact artifact, operation, or distinction is being used?
2. What documentary basis and provenance support it?
3. What scope, limitations, uncertainty, prohibited uses, and dependencies are
   declared?
4. What tests or reviews support the proposed use, including false-positive and
   adversarial considerations where applicable?
5. Which lifecycle actions does the pull request declare?
6. Did required CI pass?
7. Which authenticated owner merge authorized the resulting state?
8. Which version-pinned Manifest and commit govern each production run?

## IX. EFFECT ON DOCUMENT 000012

1. The conceptual distinctions protected by Section XI remain in force; no
   automated check silently converts one state into another.
2. One explicit owner merge may nevertheless authorize several declared states
   together. Joint authorization is not a silent conversion.
3. Section XI.6 applies to protected definition changes. Routine conformity
   does not require repetitive specialized-authority determinations.
4. Section XII.3 is satisfied by the accompanying founder ratification.
5. Future constitutional changes remain in the protected lane and continue to
   require AMD identity and explicit owner approval.

## X. MANIFEST AND ACTIVATION RULE

1. GitHub remains canonical; Neo4j remains a derived, rebuildable projection.
2. A production Manifest shall identify exact component versions, repository
   commit, dependencies, exclusions, permitted use, and fixity information.
3. The Manifest may be created or updated in the same routine pull request that
   integrates its components.
4. Owner merge releases the Manifest and authorizes its declared components for
   later production runs.
5. No merge mutates the Manifest or inputs of an execution already underway.

## XI. NON-NEGOTIABLE SAFEGUARDS

This Amendment does not authorize:

1. silent rewriting or erasure of canonical history;
2. canonical identifier reuse;
3. loss of documentary provenance, uncertainty, limitations, or scope;
4. treatment of interpretation, inference, or hypothesis as documentary fact;
5. engine or LLM self-authorization;
6. treating CI success, graph structure, or model confidence as substantive
   proof;
7. direct Neo4j mutation as a change to canonical repository meaning; or
8. mutation of a version-pinned active run.

## XII. TRANSITION

1. REG-000000003 remains preserved as optional infrastructure. Its admission
   creates no mandatory candidacy prerequisite, and routine changes need not
   use it.
2. HOC-000000001 may proceed directly through a single routine pull request
   that declares its intended validation, integration, Manifest, and activation
   effects and satisfies this Amendment.
3. Existing records need not be rewritten solely to conform to the simplified
   procedure.
4. Future operational packages should create only records that carry distinct
   evidentiary or operational value.

## XIII. FINAL DECLARATION

Repository governance shall be proportional to risk. Routine work requires one
coherent proposal, sufficient evidence, passing CI, and one authenticated owner
merge. Protected changes retain explicit scrutiny. Documentary and epistemic
integrity remain mandatory; serial permission loops do not.

**END OF AMENDMENT**
