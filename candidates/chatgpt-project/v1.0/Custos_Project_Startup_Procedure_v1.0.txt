CUSTOS PROJECT STARTUP PROCEDURE
Version 1.0
Prepared: 2026-07-22

Purpose: Start or resume a Custos production session in a new ChatGPT Project without relying on conversational memory or reopening settled architecture.

======================================================================

I. ONE-TIME PROJECT SETUP

1. Create the new ChatGPT Project for Custos.
2. Install Custos_ChatGPT_Project_Instructions_v1.0.txt as the Project instructions.
3. Add Custos_Codex_Complete_v1.0.txt as a Project reference file.
4. Make the GitHub repository izzy9118-blip/custos available through the Project’s authorized repository access or a current local clone.
5. Add source witnesses only when they are needed for the active production unit. Their presence in the Project does not make them canonical.

Neo4j, a graph projection, and an external-AI adapter are not required for ordinary file-based corpus production.

======================================================================

II. SESSION BOOT

At the beginning of every repository-production session, the AI shall perform the following checks before reporting readiness:

1. REPOSITORY IDENTITY
   - Confirm the target is izzy9118-blip/custos.
   - Record the current branch and HEAD commit.
   - Inspect the working tree and preserve unrelated changes.

2. GOVERNING AUTHORITY
   - Read applicable repository instructions.
   - Locate and read the current Custos Codex if the repository contains an admitted version.
   - Locate the active manifest, release record, or equivalent commit-pinning authority when one exists.
   - Distinguish Project instructions from canonical repository instruments.

3. CHECKPOINT RECOVERY
   - Determine the latest artifact and state actually evidenced by Git history and repository records.
   - Distinguish proposed, produced, validated, committed, admitted, and certified work.
   - Compare that evidence with the user’s stated or remembered checkpoint.
   - Disclose any conflict; do not silently choose or rewrite a record.

4. ACTIVE PRODUCTION UNIT
   - Identify the next authorized Citation Object or supporting source-recovery task.
   - Read only the governing and evidentiary files needed for that unit.
   - Establish whether the documentary boundary and fixed witnesses are sufficient to begin.

5. EXECUTION DECISION
   - If sufficient: begin production immediately.
   - If partly sufficient: complete every safe clerical or documentary step available, preserve it, then issue a lawful stop for the exact remainder.
   - If access or evidence is absent: identify the precise missing item and the exact action that would unblock production.

Readiness is established by these checks, not by the existence of a boot prompt.

======================================================================

III. STANDARD SESSION OPENING PROMPT

Paste this as the first message of a new production chat:

Resume Custos production from the canonical repository izzy9118-blip/custos. Execute the installed Custos Startup Procedure before reporting status. Inspect the current repository state and the files relevant to the active production unit, reconcile them with any stated checkpoint, identify the next authorized unit, and begin it immediately. Do not redesign the system. End only with verified production or a precise lawful stop.

======================================================================

IV. TASK-SPECIFIC OPENING PROMPT

When the intended unit is already known, use:

Resume Custos production from izzy9118-blip/custos. The active task is: [INSERT EXACT TASK]. Verify the repository checkpoint and evidence for this task, then execute the next safe unit under the Custos Codex. Preserve unrelated work and immutable history. Report exact files, validation, repository evidence, and any lawful blocker. Do not stop at a plan while authorized work remains possible.

======================================================================

V. REQUIRED FIRST STATUS

The first substantive status after inspection shall state only:

- repository branch and HEAD;
- whether the working tree contains relevant or unrelated changes;
- governing version or manifest actually found;
- last evidenced production state;
- active unit and whether its evidence is sufficient;
- the action now being executed, or the precise lawful blocker.

The AI shall not claim that the system is ready merely because files, schemas, or prompts exist.

======================================================================

VI. REQUIRED SESSION CLOSE

Every session shall close with:

1. Outcome: verified production or lawful stop.
2. State reached: proposed, produced, validated, committed, admitted, or certified.
3. Exact artifact paths.
4. Validation or test results.
5. Commit SHA or equivalent canonical evidence, if a repository write occurred.
6. Preserved uncertainties and blockers.
7. The next authorized production unit.

No completion claim may exceed the evidence reported here.

======================================================================

VII. CURRENT TRANSITION NOTE

The new ChatGPT Project is a working successor environment. The former Custos system remains historical evidence. The first session in the new Project must determine the live repository checkpoint from GitHub rather than assuming that any checkpoint stated in this file is still current.
