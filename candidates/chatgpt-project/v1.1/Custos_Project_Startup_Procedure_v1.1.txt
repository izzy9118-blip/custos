CUSTOS PROJECT STARTUP PROCEDURE
Version 1.1
Prepared: 2026-07-22

Purpose: Start or resume a Custos session with a verified publication path and
with shared textual examination as the center of inquiry.

This version prospectively supersedes v1.0 for ChatGPT Project operation. The
v1.0 file remains preserved as the procedure under which the demonstrated
authorization and shared-inquiry defects occurred.

======================================================================

I. ONE-TIME PROJECT SETUP

1. Install the text from Custos_ChatGPT_Project_Instructions_v1.1.txt as the
   Project instructions.
2. Keep Custos_Codex_Complete_v1.0.txt as a Project reference file.
3. Add Custos_Codex_Amendment_001_Shared_Textual_Examination_v1.0.txt as the
   prospective amendment to Rules I and V.
4. Replace Custos_Project_Startup_Procedure_v1.0.txt with this v1.1 procedure,
   while preserving v1.0 outside the active Project sources or as clearly
   superseded history.
5. Authorize the GitHub integration for izzy9118-blip/custos and select that
   repository explicitly.
6. Add source witnesses only when required for the active unit. Upload does not
   confer canonical status.

Neo4j, a graph projection, and an external-AI adapter are not required for
ordinary file-based corpus production or shared reading.

======================================================================

II. SESSION BOOT

Before reporting readiness, perform these checks in order.

1. REPOSITORY IDENTITY
   - Confirm the target is izzy9118-blip/custos, not a similarly named repo.
   - Record the default branch and its HEAD.
   - Record the working branch and HEAD when one exists.
   - Inspect relevant changes and preserve unrelated work.

2. PUBLICATION CAPABILITY
   - Verify read access and write access separately.
   - Do not infer write authority from public visibility or successful reads.
   - For authorized changes, create or confirm the working branch from the
     verified base commit before substantive production.
   - Never modify main directly unless the user expressly orders it.

   If remote writing is unavailable, establish durable noncanonical staging
   before substantive production. The staged package must preserve:

   - repository and base commit;
   - intended branch and repository paths;
   - complete candidate files or an applicable patch;
   - source and artifact hashes;
   - validation results; and
   - the exact publication blocker.

   When write access returns, publish that preserved candidate rather than
   regenerating it.

3. GOVERNING AUTHORITY
   - Read applicable repository instructions and current admitted instruments.
   - Read Custos_Codex_Complete_v1.0.txt and Amendment 001 as Project authority.
   - Where v1.0 and Amendment 001 conflict, apply Amendment 001 prospectively.
   - Distinguish Project authority from canonically admitted repository law.

4. CHECKPOINT RECOVERY
   - Determine the latest artifact and state evidenced by Git history.
   - Distinguish proposed, documentary preparation, ready for shared
     examination, shared examination in progress, produced, validated,
     committed, admitted, and certified.
   - Compare repository evidence with the user's checkpoint.
   - Disclose conflicts; never silently select or harmonize a record.

5. ACTIVE INQUIRY UNIT
   - Identify the complete Strauss passage and note governing the active unit.
   - Identify which witnesses are fixed and which remain missing or limited.
   - Determine the actual phase:
     A. documentary preparation;
     B. ready for shared examination;
     C. shared examination in progress;
     D. preservation and validation; or
     E. lawful stop.

6. EXECUTION DECISION
   - If documentary preparation remains, complete the safe necessary portion.
   - If the texts are ready, do not silently continue into a finished report;
     present the Shared Examination Packet.
   - If shared examination has occurred, preserve it accurately and continue
     with validation or authorized repository work.
   - If evidence or access is absent, preserve available work and issue a
     precise lawful stop.

Readiness is established by these checks, not by the existence of a prompt,
schema, successful read, or repository connection.

======================================================================

III. SHARED EXAMINATION PACKET

When the unit is ready, present a concise packet in the conversation containing:

1. Strauss's exact anchor and sufficient immediate context.
2. The complete note.
3. The cited passages and sufficient local context.
4. Edition, translation, OCR, provenance, and witness limits material to
   interpretation.
5. The central question Strauss places before the reader.
6. Two to four open comparative questions without preemptive answers.
7. One opening question addressed to the user.

Do not replace this packet with a status report or a completed interpretation.
Do not overwhelm the shared reading with every clerical field. Bring forward
only what the present textual problem requires; preserve the full apparatus in
the repository candidate.

Waiting for the user's answer is a required active stage. It is not a lawful
stop. Continue by testing readings against the words, following citations,
offering alternatives as proposals, and preserving real disagreement.

======================================================================

IV. PRESERVING THE SHARED INQUIRY

After the examination, update or create the Citation Object record with:

- the first encounter with Strauss before the citation was followed;
- the relevant documentary preparation;
- the independent reconstruction of each cited author;
- the comparative return and reader transformation;
- documented findings and classified interpretive propositions;
- user readings, AI proposals, and jointly tested judgments distinguished by
  provenance;
- unresolved disagreement and uncertainty;
- witness limits and future documentary needs; and
- the exact completion or continuing-state declaration.

Do not treat conversation as canonical merely because it occurred. The user
shall have an opportunity to correct the preserved account before admission or
certification is proposed.

======================================================================

V. STANDARD SESSION OPENING PROMPT

Use this as the first message of a new production chat:

Resume Custos from the canonical repository izzy9118-blip/custos. Execute the
installed v1.1 Startup Procedure before reporting readiness. Verify read and
write capability separately, recover the exact repository checkpoint, and
identify the phase of the active Citation Object. Complete any necessary
documentary preparation. When the texts are sufficient, place the bounded
Shared Examination Packet before me and begin examining Strauss with me; do not
turn it into a finished interpretive report first. Preserve completed inquiry
on a working branch, never on main directly. End repository action only with
verified production or a precise lawful stop.

======================================================================

VI. TASK-SPECIFIC OPENING PROMPT

When the intended unit is known, use:

Resume Custos from izzy9118-blip/custos. The active task is: [INSERT EXACT
TASK]. Verify the branch, base commit, write path, governing instruments,
documentary boundary, fixed witnesses, and present phase. Perform necessary
documentary preparation, then examine the central textual problem with me under
the Shared-Examination Gate before recording a comparative conclusion. Preserve
unrelated work and immutable history. Report exact files, validation, commit
evidence, and any precise blocker.

======================================================================

VII. REQUIRED FIRST STATUS

The first status after inspection shall state only:

- repository, working branch, base SHA, and HEAD;
- read capability and write capability as separate results;
- relevant or unrelated changes;
- governing versions actually found;
- last evidenced artifact state;
- active Citation Object and current phase;
- evidence sufficient for the next phase; and
- the documentary action being executed or the Shared Examination Packet now
  being prepared.

Do not claim readiness merely because files, schemas, prompts, or public reads
exist.

======================================================================

VIII. REQUIRED SESSION CLOSE

Every repository-production session shall close with:

1. Outcome: verified production, active shared-examination checkpoint, or
   lawful stop.
2. State reached.
3. Exact artifact paths.
4. Validation or test results.
5. Branch and commit SHA if a repository write occurred.
6. Preserved readings, disagreements, uncertainties, and blockers.
7. The next authorized documentary or shared-inquiry unit.

No completion claim may exceed the evidence or the shared examination actually
recorded.
