# STRAUSSIAN DOCUMENTARY MEMORY
# COGNITIVE MEMORY REGISTER AND MANIFEST SPECIFICATION

**Version:** 1.0  
**Classification:** Repository Specification  
**Status:** Finalized for Repository Admission  
**Identifier:** Not assigned  
**Purpose:** Define how canonical artifacts become authorized reusable cognition without creating a new ontology class or identifier family

## I. CONSTITUTIONAL BASIS

This specification is subordinate to:

- Documentary Ontology;
- Identifier Specification;
- Relationship Specification;
- Controlled Vocabulary Specification;
- Constitutional Admission Protocol;
- Repository Cognitive Architecture;
- Inquiry Engine Authority Specification, upon admission;
- Repository Safeguards;
- Repository Epistemic Hierarchy.

If conflict occurs, the higher governing document prevails.

## II. GOVERNING DISTINCTION

The Repository distinguishes:

1. **Preserved artifact** — exists in Documentary or Inquiry Memory.
2. **Candidate Cognitive Component** — designated for possible reuse.
3. **Certified component** — has passed specified certification.
4. **Integrated Cognitive Component** — authorized for production use.
5. **Active component** — included in the Manifest governing a particular engine run.

These states shall not be collapsed.

## III. NO NEW ONTOLOGY CLASS

Candidate Cognitive Component, Certified Inquiry Procedure, Integrated Cognitive Component, and Cognitive Memory are not new ontology classes.

Every component retains one native canonical class recognized by the Documentary Ontology.

Possible native classes include:

- Claim;
- Interpretation;
- Structural Pattern;
- Hermeneutic Object;
- Inquiry Architecture Record;
- Safeguard;
- Specification;
- another existing class where warranted.

A designation never replaces native class identity.

## IV. CANDIDATE COGNITIVE COMPONENT

A Candidate Cognitive Component is a canonical artifact or entity designated for possible reuse in future inquiry.

Candidate designation requires:

- canonical identifier;
- native ontology class;
- source inquiry;
- documentary basis;
- reason for candidacy;
- intended scope;
- intended engine use;
- known limitations;
- Documentary Decision Record;
- Candidate Cognitive Component Register entry.

Candidate designation confers no production authority.

A noncanonical final may be evaluated before admission, but it cannot enter the Candidate Register until it is admitted in its native class.

## V. CERTIFIED INQUIRY PROCEDURE

A Certified Inquiry Procedure is a reusable ordered investigative procedure represented as an Inquiry Architecture Record.

Required elements:

- purpose;
- triggering conditions;
- required inputs;
- ordered stages;
- permitted operations;
- prohibited operations;
- evidence requirements;
- branch conditions;
- failure conditions;
- termination conditions;
- positive validation cases;
- negative validation cases;
- adversarial validation cases;
- version history;
- Validation Records;
- Certification Record;
- integration status.

A procedure is certification pending merely because it appears in a synthesis or process register.

## VI. LITERARY AND HERMENEUTIC COMPONENTS

A literary technique derived from the Hermeneutic Object Taxonomy shall be represented in its native class, ordinarily:

- Interpretation; or
- Structural Pattern.

Each component shall preserve:

- Strauss’s terminology;
- exact source location;
- primary passage;
- mechanism;
- examples;
- investigative requirement;
- status of reconstruction;
- uncertainty;
- distinctions from related techniques;
- permitted engine use;
- prohibited inference;
- version history.

The Hermeneutic Object remains authoritative over machine-readable projections.

## VII. VALIDATION

Validation shall test, as applicable:

- documentary traceability;
- fidelity to the source inquiry;
- reproducibility;
- internal coherence;
- relation to constitutional safeguards;
- scope;
- false-positive risk;
- positive examples;
- negative examples;
- ambiguous examples;
- adversarial examples;
- cross-work portability;
- cross-author portability;
- termination behavior;
- interaction with rival components.

Validation results are preserved in Validation Records.

Validation does not confer certification.

## VIII. CERTIFICATION

Certification requires an authorized Certifier.

The Certification Record shall identify:

- component;
- native ontology class;
- validated version;
- validation records;
- documentary basis;
- certified scope;
- excluded scope;
- permitted use;
- prohibited use;
- dependencies;
- unresolved limitations;
- certifying authority;
- date;
- result.

Certification does not by itself activate the component for engine use.

## IX. INTEGRATION DECISION

Integration requires a Documentary Decision Record.

The decision shall state:

- component identifier;
- version;
- certification record;
- integration status;
- effective date;
- scope;
- dependency requirements;
- incompatibilities;
- superseded components, if any;
- withdrawal conditions;
- responsible authority.

A certified component becomes integrated only after entry into the Cognitive Memory Register.

## X. COGNITIVE MEMORY REGISTER

The Cognitive Memory Register is the authoritative collection of integrated components.

Each entry shall contain:

- register-entry identifier or stable entry key;
- component canonical identifier;
- native ontology class;
- component label;
- component version;
- component type:
  - literary distinction;
  - structural pattern;
  - inquiry procedure;
  - evidentiary test;
  - inferential constraint;
  - termination rule;
  - source-selection rule;
  - other approved type;
- source inquiry identifiers;
- validation record identifiers;
- certification record identifier;
- integration decision identifier;
- lifecycle status;
- certified scope;
- permitted use;
- prohibited use;
- dependencies;
- effective date;
- version history;
- audit history.

Register history is append-preserving.

Withdrawal or supersession does not delete prior entries.

## XI. CANDIDATE COGNITIVE COMPONENT REGISTER

The Candidate Register preserves designated candidate components awaiting validation, certification, rejection, or withdrawal.

Each entry shall include:

- component canonical identifier;
- native class;
- source inquiry;
- candidacy decision;
- intended scope;
- intended use;
- validation status;
- current outcome;
- audit history.

Rejected candidates remain preserved.

## XII. COGNITIVE MEMORY MANIFEST

A Cognitive Memory Manifest is the structured declaration of the exact integrated components authorized for a class of engine execution or a particular release.

Each Manifest shall include:

- Manifest identifier;
- version;
- release status;
- Repository commit;
- governing specification versions;
- Taxonomy source version;
- DF-derived Inquiry Architecture version;
- included component identifiers and versions;
- component certification records;
- component integration decisions;
- dependency graph;
- exclusions;
- known conflicts;
- permitted engine mode;
- build date;
- responsible authority;
- fixity information;
- predecessor Manifest;
- changelog.

A Manifest shall be immutable after release.

A correction creates a new Manifest version.

## XIII. MANIFEST RELEASE RULES

A Manifest may be released only when:

- every included component resolves;
- every component is in the Cognitive Memory Register;
- every component version matches its certification;
- every integration decision resolves;
- dependencies are complete;
- no withdrawn component is included as active;
- no unresolved conflict blocks release;
- fixity and version data are complete;
- Responsible Authority approves release.

## XIV. ENGINE USE

At run initialization, the Engine records the selected Manifest.

The Manifest is frozen during execution.

The Engine may not activate:

- a candidate component;
- a certified but unintegrated component;
- an integrated component omitted from the selected Manifest;
- a later component added after run initialization.

## XV. DEVELOPMENT MODE

Development Mode may assemble an explicit Development Manifest containing provisional components.

A Development Manifest:

- is noncanonical or clearly nonproduction;
- must identify every provisional component;
- may not be mistaken for a Cognitive Memory Manifest;
- may produce only provisional outputs;
- confers no admission or certification authority.

## XVI. EXISTING HERMENEUTIC PROCESS REGISTER

Existing Hermeneutic Process Register entries remain preserved as historical Inquiry Memory and candidate process material.

They shall not be silently converted into Cognitive Memory.

For possible acquisition, each entry must:

1. identify its native ontology representation;
2. identify supporting inquiries;
3. be admitted canonically where required;
4. receive candidacy decision;
5. undergo validation;
6. undergo certification;
7. receive integration decision;
8. enter the Cognitive Memory Register;
9. appear in a released Manifest.

## XVII. CORRECTION, SUPERSESSION, AND WITHDRAWAL

A component may be corrected, superseded, or withdrawn through governed records.

Rules:

- prior versions remain preserved;
- a corrected component must be recertified when the certified meaning or operation changes;
- a superseding component receives its own appropriate identity where required;
- a withdrawn component is removed only from future Manifests;
- prior runs remain reproducible under the Manifest they used;
- no past Manifest is silently rewritten.

## XVIII. NEO4J PROJECTION

Neo4j may project:

- Candidate Register entries;
- Cognitive Memory Register entries;
- Manifest contents;
- source inquiry links;
- validation links;
- certification links;
- integration decisions;
- component dependencies.

Every projected object and relation must resolve to canonical GitHub records.

Neo4j cannot activate, certify, integrate, withdraw, or correct a component.

## XIX. MINIMUM ACQUISITION TEST

A component shall not be integrated unless the Repository can answer:

1. What documentary inquiry produced it?
2. What exact operation or distinction is being reused?
3. What evidence warrants it?
4. What are its limits?
5. On what independent cases was it tested?
6. What false positives were considered?
7. What authority validated it?
8. What authority certified it?
9. What decision integrated it?
10. What Manifest activates it?
11. Would removing it measurably diminish the engine’s Straussian documentary fidelity?

## XX. FINAL DECLARATION

Inquiry Memory preserves what happened.

Cognitive Memory preserves only what has earned reusable authority.

Certification alone does not activate cognition.

Integration alone does not activate cognition.

The released Manifest determines what a production engine run may use.

**END OF FINAL**
